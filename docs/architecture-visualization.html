V1(기존) vs V2(목표) 구조 비교 페이지

V1 Contract Topology

flowchart LR U[Users / Operators] --> P1[DAOCommitteeProxy] P1 --> P2[DAOCommitteeProxy2\nselector router] P2 --> IA[Implementation A\nDAOCommittee_V1] P2 --> IB[Implementation B\nDAOCommitteeOwner] IA --> AM[DAOAgendaManager] IA --> DV[DAOVault] IA --> CF[CandidateFactory] CF --> C[Candidate Contracts xN] C --> IA IA --> L2R[Layer2Registry] IA --> SM[SeigManager] subgraph V1-Risk Focus S1[Storage layout consistency\nProxy + Impl + AddOn] S2[Selector routing correctness] S3[Owner/Admin control path] end P2 -.-> S2 P1 -.-> S1 IA -.-> S3

핵심: Proxy → Proxy2 → Impl 구조, selector 라우팅 및 storage 정합성 관리가 중요.

V1 Governance Flow (요약)

sequenceDiagram participant Proposer participant Gov as DAOCommittee participant Agenda as DAOAgendaManager participant Exec as Target Contracts Proposer->>Gov: onApprove / createAgenda Gov->>Agenda: newAgenda() Note over Agenda: NOTICE -> VOTING Gov->>Agenda: castVote() Note over Agenda: ACCEPT / REJECT / DISMISS Gov->>Exec: executeAgenda() if ACCEPT

V2 Contract Topology

flowchart LR Token[vTON] --> DR[DelegateRegistry] DR --> GOV[DAOGovernor] GOV --> TL[Timelock] SC[SecurityCouncil\nVeto + Pause only] --> GOV SC -.->|no direct cancel| TL GOV --> P[Proposal Lifecycle] TL --> EX[Execution Targets] subgraph Governance Safety G1[Self-defense restriction\n(SC cannot veto proposals targeting itself)] G2[SC validity window 12m\n+ permissionless expiry] G3[Burn removed in voting path\n(spec 0.1.4 alignment)] end GOV -.-> G1 SC -.-> G2 GOV -.-> G3

핵심: non-upgradeable + Timelock 중심, SC 권한 최소화(Veto/Pause), 만료 메커니즘 포함.

V2 Proposal Lifecycle

stateDiagram-v2 [*] --> Pending Pending --> Active Active --> Defeated Active --> Succeeded Succeeded --> Queued Queued --> Executed Queued --> Expired Pending --> Canceled Active --> Canceled Succeeded --> Canceled Queued --> Canceled

V1 ↔ V2 비교 핵심

  • 구조: V1은 Proxy 중심 라우팅 복잡도, V2는 Governor + Timelock 중심 단순화
  • 보안: V2는 SC 권한을 Veto/Pause로 제한하고 만료를 온체인에서 강제
  • 거버넌스: V2는 delegation 기반 투표 흐름 명확화, spec 0.1.4 기준 burn 제거 정합
  • 마이그레이션 리스크: V1의 상태 이관 정확도(members/candidates/agenda 상태) + 권한 이관(Timelock 귀속)이 핵심
