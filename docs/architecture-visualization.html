V1(기존) vs V2(목표) 구조 비교 페이지

V1 Contract Topology

flowchart LR U["Users / Operators"] -->|"TON.approveAndCall"| TONTK["TON Token"] TONTK -->|"onApprove"| P1["DAOCommitteeProxy\nstorage holder\nslots 0-25"] ADMIN["Admin\nGnosis Safe 2/3 Multisig"] -.->|"pauseProxy\nupgradeTo"| P1 P1 --> P2["DAOCommitteeProxy2\nselector router"] P2 -->|"default"| IA["Implementation A\nDAOCommittee_V1"] P2 -->|"4 selectors"| IB["Implementation B\nDAOCommitteeOwner\nsetQuorum\nsetActivityRewardPerSecond\nincreaseMaxMember\ndecreaseMaxMember"] IA --> AM["DAOAgendaManager"] IA --> DV["DAOVault\nTON/WTON custody"] IA --> CFP["CandidateFactoryProxy"] CFP --> CF["CandidateFactory"] CF --> C["Candidate Proxy xN\n1 operator = 1 proxy"] C -->|"castVote"| P1 IA --> L2R["Layer2Registry"] IA --> SM["SeigManager"] subgraph Risk ["V1 Risk Focus"] S1["Storage layout\nslots 0-25"] S2["Selector routing\ncorrectness"] S3["Admin control path"] end P2 -.-> S2 P1 -.-> S1 ADMIN -.-> S3

핵심: Proxy → Proxy2 → Impl 2단계 delegatecall 구조. Storage는 DAOCommitteeProxy의 slots 0-25에 보유.
Admin(Gnosis Safe 2/3 Multisig)이 pauseProxy, upgradeTo 권한 보유.
Candidate Proxy는 castVote 시 DAOCommitteeProxy(P1)를 경유하여 Proxy2 → Impl A로 delegatecall.
Impl B는 4개 함수만 처리: setQuorum, setActivityRewardPerSecond, increaseMaxMember, decreaseMaxMember.

V1 Governance Flow

sequenceDiagram participant Proposer participant TON as TON Token participant Gov as DAOCommitteeProxy participant Agenda as DAOAgendaManager participant Cand as Candidate xN participant Exec as Target Contracts Proposer->>TON: approveAndCall with 10 TON TON->>Gov: onApprove callback Gov->>Gov: 10 TON burn to address 1 Gov->>Agenda: newAgenda Note over Agenda: NOTICE state - 16 day minimum Note over Cand,Agenda: VOTING state - min 2 days Cand->>Gov: castVote via Proxy (1 operator = 1 vote) Gov->>Agenda: record vote Note over Agenda: 1st vote snapshots current members Note over Agenda: Quorum: YES >= 2 of 3 members Note over Agenda: WAITING_EXEC or REJECT or DISMISS Note over Gov,Exec: Execution window: 7 days Gov->>Exec: executeAgenda - atomic or non-atomic Note over Agenda: EXPIRED if not executed within 7 days

각 L2 Operator는 개별 Candidate Proxy를 통해 투표 (1인 1표).
Admin(Gnosis Safe Multisig)이 pauseProxy, 구현체 업그레이드 권한 보유.

V2 Contract Topology

flowchart LR subgraph Token Layer TON["TON\nexternal ERC20"] vTON["vTON\nERC20Votes + Halving\nMax 100M / Epoch 5M"] TON -->|"10 TON burn to 0xdead\nproposal cost"| GOV end subgraph Delegation Layer DR["DelegateRegistry\nOwnable + ReentrancyGuard"] Holders["vTON Holders"] -->|"delegate / undelegate\nredelegate"| DR DEL["Registered Delegates"] -->|"register / update\ndeactivate / reactivate"| DR end vTON -->|"custody transfer"| DR subgraph Governance Layer GOV["DAOGovernor\nOwnable + Pausable + ReentrancyGuard"] DR -->|"getVotingPower\nisRegisteredDelegate"| GOV DEL -->|"castVote\ncastVoteWithReason"| GOV end subgraph Execution Layer TL["Timelock\nReentrancyGuard\ndelay 7-30d / grace 14d"] GOV -->|"queue / execute\ncancelTransaction"| TL TL -->|"call target"| EX["Execution Targets\nvTON / DR / GOV / SC / ..."] end subgraph Emergency Layer SC["SecurityCouncil\nMultisig 2/3 threshold\n12m validity + expiry"] SC -->|"cancelProposal"| GOV SC -->|"pause / unpause"| GOV end EX -.->|"addMember / removeMember\nrenewCouncilValidity"| SC

핵심: non-upgradeable 구조. TON은 기존 외부 ERC20 토큰 (V2 자체 컨트랙트 아님).
vTON Holder는 직접 투표 불가 (vTON.delegate() 호출 시 UseDelegateRegistry 에러) — DelegateRegistry를 통해 Delegate에게 위임 후 Delegate가 투표.
Timelock이 모든 실행을 중개하며, SecurityCouncil은 Veto/Pause 전용 권한만 보유. SC 멤버 변경은 DAO 거버넌스(Timelock)를 통해서만 가능.

V2 Proposal Lifecycle

flowchart TD S0(("Start")) --> Pending Pending -->|"votingDelay 1d"| Active Active -->|"quorum or passRate fail"| Defeated Active -->|"quorum and passRate pass"| Succeeded Succeeded -->|"queue"| Queued Queued -->|"execute after eta"| Executed Queued -->|"grace period over"| Expired Pending -->|"proposer"| Canceled Active -->|"proposer"| Canceled Succeeded -->|"guardian"| Canceled Queued -->|"guardian veto"| Canceled

votingDelay = 7,200 blocks (~1d), votingPeriod = 50,400 blocks (~7d), timelockDelay = 7 days, grace = 14 days.
ProposalGuardian은 SecurityCouncil과 별도 역할 — setProposalGuardian()으로 임의 주소 설정 가능 (SC 설정은 일반적이나 필수 아님).
ProposalGuardian에는 self-defense restriction 적용 (자기 자신을 대상으로 하는 프로포절은 취소 불가).

V2 Roles & Permissions

flowchart TB subgraph Gov ["DAO Governance"] GOV_NODE["DAOGovernor\nOwner: Timelock"] TL_NODE["Timelock\n2-step admin transfer"] end subgraph Tok ["Token"] VTON_OWN["vTON Owner"] -->|"setMinter\nsetEmissionRatio"| VTON_CT["vTON Contract"] end subgraph Guardians ["Guardian Roles"] PG_NODE["PauseGuardian"] -->|"pause\nunpause"| GOV_NODE PROP_NODE["ProposalGuardian"] -->|"cancel non-final proposals\nself-defense restriction"| GOV_NODE end subgraph SCG ["Security Council"] SC_MEM["SC Members\nthreshold: 2 of 3"] -->|"cancelProposal\npauseProtocol\nunpauseProtocol"| SC_CT["SecurityCouncil\n12m validity"] end GOV_NODE -->|"setQuorum\nsetPassRate\nsetVotingDelay\nsetVotingPeriod\nsetMaturityPeriod\nsetProposalThreshold\nsetProposalCreationCost\nsetProposalGuardian\nsetPauseGuardian"| PARAMS["Governor Params\nowner-only"] TL_NODE -->|"addMember\nremoveMember\nrenewValidity\nsetThreshold"| SC_CT TL_NODE -->|"setDelay\nsetGovernor\nsetSecurityCouncil"| TL_NODE2["Timelock Params\nadmin-only"]

모든 파라미터 변경은 DAOGovernor Owner(= Timelock)를 통해서만 가능.
ProposalGuardian과 SecurityCouncil은 별도 역할 — SC를 ProposalGuardian으로 설정 가능하나 필수는 아님.
Timelock Admin은 2-step transfer(setPendingAdmin → acceptAdmin)로 이전.
SC는 긴급 대응(Veto/Pause)만 가능하며, 12개월 후 permissionless하게 만료 가능.

마이그레이션 핵심 고려사항

  • 진행 중 Agenda 처리: V1에 NOTICE/VOTING/WAITING_EXEC 상태인 agenda가 있을 경우, V1에서 완료 후 전환하거나 V1 pause 후 마이그레이션 필요
  • DAOVault 소유권: DAOCommitteeProxy → Timelock으로 treasury 관리 권한 이전
  • Admin 권한 이전: Gnosis Safe의 V1 관리 권한을 V2 Timelock 기반으로 전환. 전환 기간 동안 V1 pauseProxy + V2 Timelock 이중 관리 가능성
  • TON burn 경로: V1은 address(1)로, V2는 0xdead로 burn — 동일 TON 토큰 사용
  • SeigManager 연동: V1 SeigManager가 V2 vTON의 minter로 등록되어야 함. Staking 레이어(DepositManager/Coinage)는 변경 없음
  • 위원회 → Delegate 전환: V1 members 개념이 V2에는 없음. 기존 operator들은 V2에서 Delegate로 등록 후 vTON 위임 필요
  • Activity Reward: V1의 activityRewardPerSecond + claimedTimestamp 메커니즘은 V2에 없음. 별도 보상 체계 필요
  • 역사적 데이터: V1의 16개 완료 agenda는 온체인에 남아 있으며, V2 UI에서 읽기 전용으로 표시 가능
