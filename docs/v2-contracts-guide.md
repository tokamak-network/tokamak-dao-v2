# Tokamak DAO v2 컨트랙트 이해 가이드

> **대상 독자**: V2 컨트랙트 구조를 처음 접하는 사람. 각 컨트랙트가 무슨 역할을 하고 어떻게 연결되는지 비유와 함께 설명합니다.
> **기준**: `contracts/src/`의 실제 소스 코드 + [contract-spec.md](../contracts/contract-spec.md) (코드가 진실의 원천).
> 함수 시그니처·이벤트 등 정확한 명세는 contract-spec.md를 참고하세요.

---

## 한눈에 보기

v2는 6개의 스마트 컨트랙트가 맞물려 돌아가는 온체인 거버넌스 시스템입니다.
토큰을 가진 사람이 직접 투표하지 않고 **"위임"을 통해서만** 의사결정에 참여하며,
통과된 제안은 즉시 실행되지 않고 **"지연 장치"**를 거칩니다.

| 항목 | 값 |
|------|-----|
| 투표 토큰 | vTON (최대 1억 개, 반감기 발행) |
| 투표 방식 | 위임 전용 (홀더는 직접 투표 불가) |
| 실행 지연 | 7일 Timelock |

### 🏛️ 큰 그림 비유: 국회

- **vTON** = 국민이 가진 "표(투표권)"
- **DelegateRegistry** = 표를 맡길 "국회의원 명부"
- **DAOGovernor** = 법안을 처리하는 "국회 본회의장"
- **Timelock** = 통과된 법이 시행되기 전 "공포 유예기간"
- **SecurityCouncil** = 위급할 때 제동을 거는 "헌법재판소"
- **VoteRelayFund** = AI 에이전트 투표의 가스비를 대신 내주는 "지원 금고"

## 전체 아키텍처

6개 컨트랙트의 권한 흐름 — **누가 누구를 호출할 수 있는지**가 핵심입니다.

```
┌──────────────────────┐
│        vTON          │  투표 토큰 (ERC20Votes)
└──────────┬───────────┘
           │ approve + 토큰 잠금(lock)
           ▼
┌──────────────────────┐
│   DelegateRegistry   │  위임 관리 + 투표력 집계
└──────────┬───────────┘
           │ getTotalDelegated()로 투표력 조회
           ▼
┌──────────────────────┐
│     DAOGovernor      │  제안 · 투표 · 실행의 중심
└──────────┬───────────┘
           │ queue() / execute() — 모든 실행은 Timelock 통과
           ▼
┌──────────────────────┐      ┌──────────────────────┐
│       Timelock       │◀─────│   SecurityCouncil    │
│  7일 지연 후 실행     │ 비상  │  2/3 멀티시그          │
└──────────────────────┘ 제동  │  제안 취소 / 일시정지   │
                              └──────────────────────┘
┌──────────────────────┐
│    VoteRelayFund     │  AI 에이전트 투표 가스 대납 (보조)
└──────────────────────┘
```

**핵심 원칙 1 — 위임 없이는 투표 없다**: vTON을 아무리 많이 가져도 직접 투표할 수 없습니다. 반드시 등록된 위임자(delegate)에게 토큰을 맡겨야 하고, 투표권은 그 위임자에게만 생깁니다.

**핵심 원칙 2 — 모든 실행은 Timelock을 거친다**: DAOGovernor는 직접 다른 컨트랙트를 호출하지 않습니다. 통과된 제안조차 Timelock에 7일간 대기시킨 뒤에야 실행됩니다.

## 01. vTON — 거버넌스 토큰

`token/vTON.sol` · ERC20 + ERC20Permit + ERC20Votes

투표권의 원천이 되는 ERC20 토큰. 일반 토큰과 다른 점은 세 가지입니다.

1. **반감기 발행** — 발행량이 늘수록 새로 찍히는 양이 점점 줄어듭니다.
2. **거래 가능 · 투표해도 소각 안 됨** — 투표는 잔액 기준이지 토큰을 태우는 게 아닙니다.
3. **직접 위임(delegate) 차단** — ERC20Votes의 기본 `delegate()`를 막아두고, 반드시 DelegateRegistry를 쓰게 강제합니다 (`UseDelegateRegistry` 에러).

> 💡 **반감기 비유**: 비트코인 채굴 보상이 시간이 지날수록 반으로 줄듯, vTON도 500만 개가 발행될 때마다(=1 에폭) 새 발행 비율이 25%씩 깎입니다. 초기에 참여한 사람일수록 같은 노력으로 더 많은 토큰을 받는 구조입니다.

### 반감기 메커니즘

`mint()`가 호출되면 두 가지 비율이 곱으로 적용됩니다:

```
실제 발행량 = 요청량 × 반감기비율(halvingRatio) × 발행비율(emissionRatio)
```

| 상수 | 값 | 의미 |
|------|-----|------|
| `MAX_SUPPLY` | 100M | 최대 1억 vTON |
| `EPOCH_SIZE` | 5M | 에폭당 500만 개 |
| `DECAY_RATE` | 0.75 | 에폭마다 ×0.75 (25%↓) |

에폭 0에서 비율 1.0 → 에폭 1에서 0.75 → 에폭 2에서 0.5625 … 처럼 줄어들고, 20 에폭이면 발행이 사실상 종료됩니다. `emissionRatio`는 DAO가 0~100% 사이로 따로 조절할 수 있는 추가 손잡이입니다.

**왜 직접 위임을 막았나?** ERC20Votes의 기본 위임은 "투표력만" 넘기지만, v2는 위임 시 실제 토큰을 Registry에 잠가서 위임받은 만큼만 정확히 투표하고 악성 위임자 소각까지 가능하게 만들기 위함입니다.

## 02. DelegateRegistry — 위임 관리소

`governance/DelegateRegistry.sol`

누가 위임자인지 등록하고, 토큰을 잠가서 투표력을 집계하는 곳.

> 🏦 **비유: 표를 맡기는 신탁 금고** — 위임은 단순히 "당신을 믿어요"라고 말하는 게 아니라, 실제 vTON을 금고(Registry)에 맡기는 행위입니다. 내가 100 vTON을 위임자 A에게 맡기면 그 토큰은 Registry에 잠기고, A의 투표력이 +100 됩니다. 돌려받으려면 `undelegate`로 회수해야 합니다.

### 두 종류의 등록

1. **위임자(delegate) 등록** — 투표를 대신할 사람이 프로필·투표 철학·관심사를 적고 `registerDelegate()`로 명부에 오릅니다.
2. **위임(delegation)** — 토큰 보유자가 `delegate(위임자, 금액)`로 vTON을 맡깁니다. 이때 토큰이 `transferFrom`으로 Registry에 실제 이동·잠금됩니다.

### 위임 흐름

```
① approve   — Registry에 vTON 사용 권한 부여
② delegate  — 토큰이 Registry로 잠김 + 위임자 투표력 ↑
③ checkpoint — 블록별 투표력 기록 저장
```

`delegateWithPermit()`을 쓰면 approve 서명과 위임을 한 트랜잭션에 묶을 수 있습니다.

### 소각 정산 — 가장 까다로운 부분

DAOGovernor가 (거버넌스 제안을 통해) 악성 위임자의 토큰을 `burnFromDelegate()`로 소각하면, 그 위임자에게 토큰을 맡긴 사람들은 **비례해서 손실을 나눠 부담**합니다. 회수 시 받는 금액:

```
실제 회수액 = 요청액 × 총위임량 / (총위임량 + 총소각량)
```

예: A에게 전체 1000이 위임됐는데 200이 소각되면, 내가 100을 회수할 때 실제로는 `100 × 1000 / 1200 ≈ 83.3`만 돌아옵니다. 나머지는 소각 손실 분담분입니다.

**참고**:
- `burnFromDelegate()`는 `governor`만 호출 가능하고, 소각 토큰은 `0xdead` 주소로 보냅니다 (ERC20은 `address(0)` 전송이 막혀 있어서).
- `autoExpiryPeriod`(위임 자동 만료)는 현재 정보성 값으로, 온체인 투표력 계산에서 강제되지 않습니다.

## 03. DAOGovernor — 거버넌스의 심장

`governance/DAOGovernor.sol` · Ownable + Pausable + ReentrancyGuard + EIP712

제안을 만들고, 투표를 받고, 통과시키고, 실행을 지시하는 중심 컨트랙트.

> 🏟️ **비유: 국회 본회의장** — 법안(제안)을 상정하려면 보증금(10 TON 소각)과 최소 의석(총량의 0.25% 투표력)이 필요합니다. 투표가 끝나면 정족수(4%)를 채우고 과반(50% 초과)이어야 통과하며, 통과돼도 바로 시행되지 않고 공포 유예(Timelock)를 거칩니다.

### 제안을 만들려면 (propose)

- **10 TON 소각** — 스팸 방지용 보증금. `0xdead`로 전송됩니다.
- **투표력 0.25% 이상** — `보유 vTON + 위임받은 vTON`이 총공급량의 0.25%(25 bp) 이상.
- **burnRate는 0만 허용** — 제안자 지정 투표 소각 기능은 스펙 0.1.3에서 제거됨. 파라미터는 ABI 호환용으로만 남아 있고 0이 아니면 revert.

### 투표 (castVote)

등록된 위임자만 투표할 수 있고, **투표력은 그 시점에 위임받은 총량(`getTotalDelegated`)으로 실시간 반영**됩니다. 선택지는 찬성(For) / 반대(Against) / 기권(Abstain).

- `castVote` — 기본 투표
- `castVoteWithReason` — 사유를 남기는 투표
- `castVoteBySig` — EIP-712 서명 투표 (대리 제출 가능 → VoteRelayFund와 연계)

### 통과 판정 로직

| 조건 | 기준 | 설명 |
|------|------|------|
| ① 정족수 (Quorum) | ≥ 4% | 총 투표수 ≥ 생성 시점 총위임량의 4%. 못 채우면 Defeated |
| ② 통과율 (Pass Rate) | > 50% | 찬성 / (찬성+반대) > 50%. **기권은 분모에서 제외** |

**중요 디테일**: 정족수와 통과율 기준은 **제안 생성 시점 값으로 고정**(`totalDelegatedAtSnapshot`, `snapshotQuorum`)되어, 도중에 파라미터가 바뀌어도 진행 중인 제안에는 영향이 없습니다. 반면 개별 투표의 가중치는 투표 시점 실시간 값입니다.

## 04. Timelock — 실행 지연 안전장치

`governance/Timelock.sol`

통과된 제안이 바로 실행되지 않도록 시간을 강제하는 컨트랙트.

> 🚪 **비유: 은행 대형 출금의 예고 기간** — 중요한 변경은 "예약"만 걸어두고 7일을 기다린 뒤에야 실행됩니다. 이 기간 동안 커뮤니티는 문제를 발견하면 빠져나가거나, SecurityCouncil이 취소할 수 있습니다. 너무 늦으면(14일 유예기간 초과) 그 예약은 만료되어 무효가 됩니다.

### 트랜잭션 생애주기

```
queueTransaction   — eta = now + 7일로 예약 (governor만)
executeTransaction — eta 도달 후 ~ 유예 14일 내 실행
cancelTransaction  — SecurityCouncil/Governor가 취소
```

| 상수 | 값 | 의미 |
|------|-----|------|
| `MINIMUM_DELAY` | 7 days | 최소 지연 — 이보다 짧게 설정 불가 |
| `MAXIMUM_DELAY` | 30 days | 최대 지연 |
| `GRACE_PERIOD` | 14 days | eta 이후 이 기간 내 실행 안 하면 만료 |

**설계 포인트**: Governor의 `queue()`는 Timelock의 `delay()`를 직접 읽어 eta를 계산합니다 (별도 값을 쓰면 해시가 어긋나 `execute()` 실패). admin 권한은 2단계 이전(`setPendingAdmin` → `acceptAdmin`)으로만 넘겨 실수로 권한을 잃는 것을 막습니다.

## 05. SecurityCouncil — 비상 대응 멀티시그

`governance/SecurityCouncil.sol`

악성 제안이나 위급 상황에 제동을 거는 **거부권(Veto) 전용** 기구.

> 🚨 **비유: 비상 정지 버튼을 가진 3인 위원회** — 총 3명(재단 1 + 외부 2)으로 구성되고, 행동하려면 2/3 이상의 동의가 필요합니다. 이들은 새로운 일을 만들 수는 없고(임의 실행 불가), 오직 위험한 제안을 막거나 시스템을 멈추는 거부권만 가집니다. 권한 남용을 막는 비대칭 설계입니다.

### 할 수 있는 일

- **제안 취소** — `cancelProposal()` (Governor의 `proposalGuardian`으로 지정되어 있어야 함, 자신을 대상으로 하는 제안은 취소 불가)
- **프로토콜 일시정지/재개** — `pauseProtocol()` / `unpauseProtocol()`
- 모든 긴급 액션은 **제안 → 승인(2/3) → 실행** 단계로 처리, 7일(`ACTION_TTL`) 내 실행하지 않으면 만료

| 항목 | 값 | 비고 |
|------|-----|------|
| 구성 | 재단 1 + 외부 2 | |
| 임계값 | 2/3 | 올림 계산, 멤버 변동 시 재조정 |
| 유효기간 | 365일 | 만료 후 DAO가 `renewCouncilValidity`로 갱신해야 작동. 만료 시 누구나 `expireCouncil()` 호출 가능 |

**견제와 균형**: 멤버 추가/삭제·임계값 변경은 `onlyDAO`(=Timelock)만 가능합니다. 위원회 스스로 멤버를 못 바꾸고 DAO 투표를 거쳐야 하며, 반대로 위원회는 DAO의 악성 제안을 막을 수 있어 상호 견제 구조가 완성됩니다. 위원회 권한도 1년마다 갱신해야 영구 집권을 막습니다.

## 06. VoteRelayFund — 가스 대납 금고

`governance/VoteRelayFund.sol`

AI 에이전트 위임자가 가스비 걱정 없이 투표하도록 돕는 보조 컨트랙트.

위임자(특히 자동 투표하는 AI 에이전트)를 위해 ETH를 예치해 두면, 릴레이어가 `castVoteBySig` 서명 투표를 대신 제출하고 사용한 가스를 이 금고에서 환급받습니다. 에이전트는 ETH를 들고 있지 않아도 투표를 이어갈 수 있습니다.

주요 함수: `deposit(agent)`, `withdraw(amount)`, `relayVote(...)`

> 에이전트 투표 시스템 전체 흐름과 알려진 문제점·개선안은 [vote-relay-fund.md](./vote-relay-fund.md) 참고.

## 제안의 생애주기

### 8가지 상태 (ProposalState)

| # | 상태 | 의미 |
|---|------|------|
| 0 | Pending | 투표 시작 대기 |
| 1 | Active | 투표 진행 중 |
| 2 | Canceled | 취소됨 |
| 3 | Defeated | 부결 (정족수·과반 미달) |
| 4 | Succeeded | 가결 |
| 5 | Queued | Timelock 대기 |
| 6 | Expired | 유예기간 초과 만료 |
| 7 | Executed | 실행 완료 |

### 전체 흐름

1. **`propose()` — 제안 생성**: 10 TON 소각 + 0.25% 투표력 확인 → 정족수/통과율 기준 스냅샷 저장. 이 시점 블록이 `snapshotBlock`. (votingDelay=0이라 바로 Active)
2. **`castVote()` — 투표 진행 (~7일, 50,400블록)**: 등록 위임자들이 찬/반/기권 투표. 투표력은 그 순간 위임받은 양으로 실시간 집계.
3. **`state()` — 자동 판정**: 투표 종료 후 정족수(4%)와 통과율(>50%)을 모두 만족하면 Succeeded, 아니면 Defeated.
4. **`queue()` — Timelock 큐잉**: 가결된 제안의 각 액션을 Timelock에 등록. `eta = now + 7일` → Queued.
5. **`execute()` — 실행 (7일 후 ~ 14일 유예 내)**: Timelock을 통해 targets의 함수들을 실제 호출 → Executed. 시간을 넘기면 Expired.

**중간 이탈 경로**: 제안자는 Pending/Active에서만 자신의 제안을 `cancel()`할 수 있고, proposalGuardian(=SecurityCouncil)은 최종 상태(Defeated/Expired)가 아니면 취소할 수 있습니다. 단, 가디언은 자기 자신을 대상으로 하는 제안은 취소 불가(`SelfDefenseRestricted`).

## 꼭 기억할 핵심 메커니즘

1. **🔒 위임 = 토큰 잠금** — 투표력만 넘기는 게 아니라 실제 vTON이 Registry에 잠깁니다. 위임받은 정확한 양만큼만 투표하고, 회수 전까지는 못 씁니다.
2. **🔥 악성 위임자 소각** — Governor가 `burnFromDelegate`로 위임 토큰을 태울 수 있고, 손실은 위임자들이 비례 분담합니다. 책임 있는 위임을 유도합니다.
3. **📉 반감기 발행** — 500만 개마다 발행 비율이 25%씩 감소. 초기 기여자에게 유리하고 인플레이션을 자연 억제합니다.
4. **⏳ 무조건 Timelock 경유** — 통과된 제안도 7일 대기. 그 사이 SecurityCouncil이 거부권을 행사할 수 있어 안전망이 이중입니다.
5. **🗳️ 실시간 투표력 · 고정 정족수** — 투표력은 투표 순간 기준(실시간)이지만, 정족수·통과율·총위임량 기준은 제안 생성 시점으로 고정됩니다.
6. **⚖️ 상호 견제** — DAO는 위원회 멤버를 바꾸고, 위원회는 DAO 악성 제안을 막습니다. 양쪽 모두 일방적 권력을 갖지 못합니다.
